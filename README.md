# AWS Cloud Security Monitoring Investigation Lab (CloudTrail+CloudWatch)

## Objective

This lab/project aims to develop foundational skills in cloud security monitoring, audit logging, and event investigation using AWS. In this lab, I configured AWS CloudTrail to capture and monitor management events across Amazon S3, EC2, and IAM services. I generated cloud activity by creating and deleting S3 buckets, launching and managing EC2 instances, and creating and removing IAM users. I then investigated the resulting CloudTrail events to identify user actions, API calls, timestamps, source IP addresses, and affected resources. Additionally, I explored Amazon CloudWatch to understand cloud monitoring capabilities and compared its role with CloudTrail in providing visibility into AWS environments. This project provided hands-on experience with AWS CloudTrail, CloudWatch, cloud auditing, security investigations, and event analysis while demonstrating how cloud security teams monitor and investigate activity within AWS environments.

### Skills Learned

This project provided hands-on experience with the following cloud security, monitoring, and investigation skills:
- AWS CloudTrail deployment and configuration
- Cloud audit logging and management event monitoring
- Security event investigation using CloudTrail Event History
- Analysis of AWS API calls and user activity
- Security monitoring and visibility within AWS environments
- Cloud security auditing and forensic investigation techniques

### Tools Used

- AWS Free Tier Account
- AWS Management Console
- AWS CloudTrail
- Amazon CloudWatch
- Amazon S3
- Amazon EC2
- AWS Identity and Access Management (IAM)

### Part 2: Enabling CloudTrail

Now that everything had been confirmed, I could now start the real work with CloudTrail. I first had to create a Trail that would be used. A trail in CloudTrail is essentaily a collection mechanism for AWS events, kind of like system or security logs. To create one, I searched "CloudTrail" and opened up the dashboard. I then clicked trails in the left menu, then selected "Create trail" (IM8). I then needed to configure this new trail. I named the trail "week7-cloudtrail" and for the storage, I created a new bucket. This bucket would store the logs that are generated by CloudTrail. I also enabled Multi-Region trail as well (IM9). One the next page I needed to configure what type of events the trail would record. I chose management events as this would record and API calls and events that would happen within AWS (IM10). I then finished that up and created the trail (IM11&12).

### Part 3: Exploring Existing Events in CloudTrail

Before intentionally generating events in CloudTrail, I wanted to see what events already existed in there. I recently created this AWS account, so there wouldn't be much events, but to my suprise, there was actually a decent amount of events generated, as CloudTrail records almost every action. To find previous events, I went to the "Event History" tab on the left menu. I wanted to see if CloudTrail recorded anytime a user logged into the console. To do this, I changed the Lookup attribute to "Event name" and searched "ConsoleLogin." After searching that, a few events popped up with that Event name (IM13). I opened one of them, and looked into it to see what kind of data was in that event. It gave me information like the Event Name (ConsoleLogin), the Event Time (whenever I logged into the console), the username of whoever logged in (root for this specfic event), the source IP address of who logged in (154.7.68.248 in this case), and the Region this event happened in (us-east-1) (IM14).

CloudTrail also generates events for any kind of action with buckets in S3. To check this, I went back to CloudTrail and Event history, then searched for the Event name "CreateBucket (IM16)." It also showed the Resources referenced, which basically means what resources were involved/affected by this Event/action. You can also see the event in JSON view if neccessary (IM17).
